# FFT `fft_init_dst` Produces Incorrect Output Outside `add_eff_app` Context

## Summary

The `fft4()` function from `app_examples/fft/fft.c` produces numerically incorrect output when compiled with `effcc` in a standalone Makefile context (not via `add_eff_app`). The `fft_init_dst` bit-reversal function computes a "pair-swap" permutation instead of the correct DIT-FFT bit-reversal, causing all FFT output bins to be wrong.

## Environment

- effcc version: 26.1.1.0 (hash `d69e415`)
- E1x evaluation kit, RISC-V 32-bit scalar
- MicroPython port: `ports/e1x/` in `EfficientComputer/micropython`
- Compilation: standalone Makefile using `$(EFFCC_DIR)/bin/effcc` with `--target=e1x -Os`

## Root Cause Analysis

`fft_init_dst` in `fft.c` uses `__builtin_elementwise_bitreverse` to compute bit-reversed addresses:

```c
int highShiftAmt = 32 - log2Size + 1;   // For N=4096: 21
int lowShiftAmt  = 32 - log2Size - 1;   // For N=4096: 19

uint32_t reversed = __builtin_elementwise_bitreverse(i);
uint32_t reversedA = reversed & 0xAAAAAAAA;  // odd-position bits
uint32_t reversedB = reversed & 0x55555555;  // even-position bits
reversedA = reversedA >> highShiftAmt;
reversedB = reversedB >> lowShiftAmt;
uint32_t srcIdx = reversedA | reversedB;
```

**`__builtin_elementwise_bitreverse` is working correctly** on E1x — verified that `bitrev(1) = 0x80000000`, `bitrev(2) = 0x40000000`, `bitrev(3) = 0xC0000000`.

The bug is in the shift amounts. For N=4096 (log2Size=12), correct 12-bit reversal requires both shifts to be `32 - log2Size = 20`:

| Index i | Expected `srcIdx` (12-bit reversal) | Got `srcIdx` (shift=21/19) |
|---------|--------------------------------------|---------------------------|
| 1       | 2048 (bit 11)                        | 1024 (bit 10)             |
| 2       | 1024 (bit 10)                        | 2048 (bit 11)             |
| 4       | 512  (bit 9)                         | 256  (bit 8)              |

The result is a "pair-swap" bijection (adjacent bit-pairs swapped) rather than a true bit reversal. This is a valid permutation with no collisions, but it is not the DIT-FFT input ordering, so all butterfly stages produce wrong output.

Changing both shifts to `32 - log2Size` produces the correct permutation (verified analytically) but does not fully fix the output, suggesting there may be an additional issue in the butterfly stages under this compilation context.

## Reproduction

**Step 1:** Build the MicroPython E1x firmware with the FFT kernel included:

```bash
cd ~/dev/projects/micropython/ports/e1x
make clean && make
eff-flash build/firmware.hex sram -p /dev/ttyACM0
```

**Step 2:** Connect to the REPL and run a DC input FFT (expected: peak at bin 0):

```python
# In minicom on /dev/ttyACM2 at 115200, or via send_file.py:
import fabric
sig = [1000] * 4096
power = fabric.fft_power(sig)
print("power[0]:", power[0])
print("peak bin:", fabric.argmax(power))
```

**Expected output:**
```
power[0]: ~1000000   (DC component, ~1000^2)
peak bin: 0
```

**Actual output:**
```
power[0]: 817216
peak bin: 3818       # wrong — should be 0
```

**Step 3:** Confirm via SDK self-test (run after Step 1):

```python
# The selftest compares fft4(sample_input) against expected.c (generated by KissFFT)
# Returns [pass=1/0, max_error, first_fail_bin]
# Expected: [1, <10, -1]
# Actual:   [0, 4314, 1]
```

The max element error of 4314 far exceeds the SDK's stated tolerance of 10.

## Additional Notes

- **The SDK's own standalone binary presumably passes its test** via `add_eff_app`/`eff_subtarget_compile_options`. The bug may be context-specific — something in the `add_eff_app` build pipeline (optimization flags, link-time behaviour, or a different version of `fft.c`) may produce correct results.
- Compiling `fft.c` with `-O0 -fno-lto`, `-Os`, or with/without `-DUSE_PRAGMAS` all produce the same wrong output, ruling out optimization level as the cause.
- The bug was discovered while porting the FFT example into the MicroPython Makefile build for `EfficientComputer/micropython`.

## Suggested Investigation

1. Build `app_examples/fft` with `add_eff_app` (scalar target) and verify it still prints `PASS` on hardware.
2. Dump the compiled `srcIdx` values for `i=1,2,3` from both build paths and compare.
3. Check whether the `fft.c` in the SDK repo has different shift amounts than what is in `app_examples/fft/fft.c`.
